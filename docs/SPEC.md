# MatchMaker 仕様書

## 1. 概要

### 1.1 アプリケーション名
**MatchMaker**(1v1 / 2v2 / 3v3 混合対応版)

### 1.2 目的
1対1 / 2対2 / 3対3 の試合を **1 つのラウンドに混在** させて組み立てられる Web アプリケーション。コート数や人数に応じて柔軟に編成できる。

### 1.3 動作環境
- **形態**: 単一HTMLファイル(`index.html`)
- **必要環境**: モダンWebブラウザ(Chrome / Safari / Edge / Firefox)
- **対応デバイス**: スマートフォン優先(最大幅 640px)、タブレット/PCでも閲覧可
- **iOS対応**: ホーム画面追加(PWA風)に対応
- **オフライン**: ネットワーク不要(全処理クライアントサイド完結)
- **依存ライブラリ**: なし(Vanilla JS / HTML / CSS のみ)

### 1.4 データ保存
- ブラウザの **localStorage** にキー `matchmaker_v2` で全状態を JSON シリアライズして保存。
- 旧版(`matchmaker3v3` リポジトリ)のキー `matchmaker3v3_v1` も読み込み可。読み込み時に新スキーマへ自動マイグレーション。

---

## 2. 画面構成

ヘッダー + 3タブの単一ページアプリ。

| 画面ID | タブ表示名 | 役割 |
|--------|-----------|------|
| `page-players` | プレイヤー | プレイヤーの登録・出場可否切替・削除・試合数表示 |
| `page-match` | 組み合わせ | 2v2 / 3v3 のコート数設定、組み合わせ生成、最新ラウンドの表示 |
| `page-history` | 履歴 | 過去ラウンドの一覧表示 |

ヘッダー右側の `⋯` ボタンでボトムシート型メニューを開き、リセット操作を実行できる。

## 3. データモデル

すべてのデータは `state` オブジェクトに保持される。

```ts
state = {
  players: Player[],
  courtConfig: { count1v1: number, count2v2: number, count3v3: number },   // 各 0..20
  rounds: Round[],                                       // 先頭が最新
}

Player = {
  id: string,            // 内部ID(時刻+乱数で生成)
  name: string,          // 表示名(最大20文字)
  isActive: boolean,     // 出場対象かどうか
  gamesPlayed: number,   // 通算試合数
  gamesByFormat: { "1v1": number, "2v2": number, "3v3": number },  // 形式別の試合数
}

Round = {
  roundNumber: number,           // 1始まりの連番
  matches: Match[],              // 各コートの試合
  resting: PlayerRef[],          // 休憩したプレイヤー
  timestamp: number,             // 生成時刻 (ms)
}

Match = {
  courtNumber: number,           // 1始まり (1v1 → 2v2 → 3v3 の順に番号付け)
  format: "1v1" | "2v2" | "3v3", // 試合形式
  teamA: PlayerRef[],            // 1 / 2 / 3 名
  teamB: PlayerRef[],            // 1 / 2 / 3 名
}

PlayerRef = { id: string, name: string }
```

### 3.1 旧スキーマからのマイグレーション

旧データ (`matchmaker3v3_v1`) は次のように変換される:

| 旧フィールド | 新フィールド |
|--------------|--------------|
| `courtCount: N` (旧 3v3 専用版) | `courtConfig: { count1v1: 0, count2v2: 0, count3v3: N }` |
| `courtConfig: { count2v2, count3v3 }` (1v1 追加前) | `count1v1: 0` を補完 |
| `Match` (formatなし) | `Match.format = "3v3"` を付与 |
| `Player` (gamesByFormatなし) | 過去の `rounds` を走査して各形式の出場回数を再集計し付与 |

マイグレーション後は新キー (`matchmaker_v2`) に保存する。

## 4. 機能仕様

### 4.1 プレイヤー管理(プレイヤータブ)

#### 4.1.1 プレイヤー追加
- 入力欄に名前を入力 → `追加` ボタン または Enter キーで登録。
- 空白のみは不可、最大20文字。
- 追加直後は `isActive = true`、`gamesPlayed = 0`。

#### 4.1.2 出場可否の切替
- 各プレイヤー左の丸チェック(✓)タップで `isActive` をトグル。
- 非アクティブのプレイヤーは組み合わせ生成の対象外だが、`gamesPlayed` / `gamesByFormat` / ペア履歴はそのまま保持される。
- **「休みたい」時はこちらを使う**(削除と異なり履歴を失わない)。

#### 4.1.3 プレイヤー削除(編集モード)
- 通常時は削除ボタンは非表示。
- プレイヤー一覧上部の **「編集」** ボタンで編集モードに入ると、各行に `✕` ボタンが現れる。
- 削除確認ダイアログは履歴喪失についての警告を含む(再追加しても別人扱いになる旨を明示)。
- 過去ラウンド内のプレイヤー名は表示上残る(IDで紐づくが、削除後は別の人として扱われる)。

#### 4.1.4 試合数バッジ
- 各プレイヤーに `〇 試合` のバッジ。

### 4.2 組み合わせ生成(組み合わせタブ)

#### 4.2.1 コート構成設定
- **1v1 面数** / **2v2 面数** / **3v3 面数** をそれぞれ `−` `+` で 0〜20 の範囲で設定。
- 合計が 0 のときはサマリーに `コート数を設定してください` を黄色で表示。

#### 4.2.2 サマリー表示
コート構成カード下部に状態に応じたメッセージを表示する:

| 状況 | 表示 |
|------|------|
| 合計0面 | `コート数を設定してください` (warn) |
| 必要人数 ≤ 出場対象 | `合計 N 面 / M 名で実施` |
| 必要人数 > 出場対象 | `あと X 名必要 (合計 M 名 / N 面)` (warn) |

#### 4.2.3 統計サマリー(3カラム)
- **出場対象** … `isActive = true` の人数
- **出場** … 今回出場する人数(条件を満たさなければ 0)
- **休憩** … `出場対象 − 出場`

#### 4.2.4 組み合わせ作成
- `組み合わせを作成` ボタン押下で生成(詳細は §5)。
- 以下の場合はボタン無効:
  - 合計コート数 = 0
  - 出場対象 < 必要人数(`count1v1 × 2 + count2v2 × 4 + count3v3 × 6`)

#### 4.2.5 直前ラウンドの取消
- `直前のラウンドを取消` で最新ラウンドを削除し、関係者の `gamesPlayed` を -1(0未満にはしない)。
- 確認ダイアログあり。
- ラウンド番号の振り直しは行わない(欠番が出る)。

#### 4.2.6 最新ラウンド表示
- 各コートに以下を表示:
  - `コート N` のタイトル
  - 右側に `1v1` / `2v2` / `3v3` のフォーマットバッジ(水色 / 緑 / オレンジで色分け)
  - `TEAM A` (水色 `#4cc9f0`) と `TEAM B` (ピンク `#f72585`) のメンバー一覧
  - 中央に `VS`
- 休憩者がいれば下部に `休憩 (N名)` を表示(区切りは ` / `)。

### 4.3 履歴(履歴タブ)
- 全ラウンドを **新しい順** に一覧表示。
- 各ラウンドのタイトルは `ラウンド N`。
- 各コートにフォーマットバッジ(1v1 / 2v2 / 3v3)が付く。
- 0 件のときは `📋 まだ履歴がありません` を表示。

### 4.4 メニュー(ボトムシート)
ヘッダー右上 `⋯` から開く。

| 項目 | 動作 |
|------|------|
| 結果を共有 | プレーンテキスト形式の結果を **Web Share API** で共有(LINE / メールなど)。非対応環境ではクリップボードにコピー、それも不可なら .txt をダウンロード。 |
| ダウンロード (.txt) | 結果をプレーンテキストファイル `matchmaker-YYYY-MM-DD.txt` として保存。 |
| 試合数をリセット | 全員の `gamesPlayed = 0`、`gamesByFormat` を 0、`rounds = []`。確認あり。 |
| 全データをリセット | プレイヤーを含む全 `state` を初期化(`courtConfig` は `{count1v1:0, count2v2:0, count3v3:1}` に)。確認あり。 |
| キャンセル | シートを閉じる |

#### 共有/ダウンロードの本文フォーマット

```
MatchMaker - YYYY/MM/DD

【ラウンド N】 HH:MM
コートK [format]: A1・A2・... vs B1・B2・...
休憩: 名前, 名前, ...

(以降ラウンドごとに続く)

---
集計
名前: N試合 (1v1:n, 2v2:n, 3v3:n)
...
```

- ラウンドは **古い順** に並べる(履歴タブ表示は新しい順だが、共有テキストは時系列が読みやすいよう逆順)。
- 集計は **過去ラウンドを名前ベースで走査して再集計**(`gamesPlayed` フィールドではなく)。これにより:
  - 削除→再追加されたプレイヤーも全試合がカウントされる
  - 削除済みでもう存在しないプレイヤーも、ラウンドに名前が残っていれば集計される
- 並び順は集計試合数の降順。

シート外タップでも閉じる。

## 5. マッチメイキングアルゴリズム

### 5.1 出場者の選出
1. アクティブプレイヤーを `gamesPlayed` 昇順でソート。
2. 同値はランダム順。
3. 先頭 `count1v1 × 2 + count2v2 × 4 + count3v3 × 6` 名を出場、残りを休憩に割り当て。

### 5.2 ペア履歴の集計
全過去ラウンドを走査し、以下の Map を構築:
- `teammate[(a,b)]`: 同じチームに属した回数
- `opponent[(a,b)]`: 対戦相手として組み合わさった回数

キーは ID 2 つを昇順で `|` 結合した文字列。**フォーマット (1v1 / 2v2 / 3v3) を区別せず合算** する。これにより 3v3 で多く組んだメンバーが、2v2 / 1v1 でも控えめに分けられる。1v1 は team size = 1 のため `teammate` の集計には寄与せず `opponent` のみ更新する。

### 5.3 グループ → A / B への分割(`bestSplit`)
- 引数 `players` のサイズ `n`(2 / 4 / 6)に応じて team size を `n / 2` に決定。
- C(n, n/2) 通りの分割を総当たり(1v1:2通り / 2v2:6通り / 3v3:20通り)。
- 各分割のスコア:

  ```
  Σ(同チーム内ペアのteammate回数) × 3
  + Σ(対戦相手ペアのopponent回数) × 2
  + 微小ランダム(同点を散らす)
  ```

- スコア最小の分割を採用。
- 1v1 (team size = 1) の場合、teammate 項は寄与しないため実質 opponent 履歴のみで決まる。

### 5.4 複数コート割当(`assignToCourts`)
- `courtSpecs`: コートごとの `{ format }` 配列。**1v1 → 2v2 → 3v3** の順で並ぶ。
- 出場者を **シャッフル** して先頭から各コートのサイズ分(2 / 4 / 6)を切り出し、`bestSplit` を実行。
- 各コートに割り当てた選手 `p` について、**形式偏りペナルティ** `gamesByFormat[p][そのコートの形式] × 2` を合計スコアに加算する(その形式を多く経験した選手ほどペナルティが高い → そのコートに置きにくい)。
- 全コートの合計スコアを比較し、シャッフルを **80 回試行**(コート 1 面のみのときは 1 回)、最小を採用。

### 5.5 スコア式まとめ
1ラウンド全体のスコアは下式の和で評価される(値が小さいほど良い):

```
Σ over courts:
   形式偏りペナルティ:  Σ_{p∈court}  gamesByFormat[p][format] × 2
 + チームメイト重複:    Σ_{p,q∈sameTeam} teammate(p,q) × 3
 + 対戦重複:           Σ_{p∈A, q∈B} opponent(p,q) × 2
 + 微小ランダム
```

### 5.6 設計の意図
- **チームメイト重複** は **対戦重複** より重く重み付け(× 3 vs × 2)し、「同じ味方ばかりになる」のを優先回避。
- 出場機会(総数)は `gamesPlayed` 昇順選出により均等化。
- **形式別出場回数** は `gamesByFormat × 2` のペナルティで均等化(短期的なランダム偏りを抑える)。
- 1v1 / 2v2 / 3v3 のペア履歴を統合してスコア計算するため、混合運用でも相性偏りを抑制。

## 6. UI / UX 仕様

### 6.1 デザインテーマ
- ダークテーマ。
- アクセント色 `#ff6b35`(オレンジ、3v3 識別にも使用)。
- 2v2 識別色 `#2ec27e`(`--good`、緑)。
- 1v1 識別色 `#4cc9f0`(`--teamA`、水色)。
- TEAM A は `#4cc9f0`(水色)、TEAM B は `#f72585`(ピンク)。1v1 タグの色と TEAM A の色は同じだが、表示位置が異なる(設定カード vs 試合カード内のチームラベル)ため文脈で区別される。

### 6.2 コート構成カード
3 行構成のステッパー。各行に以下を表示:
- 左: フォーマットタグ(`1v1` 水色 / `2v2` 緑 / `3v3` オレンジ)
- 右: `−` / 数字 / `+` のステッパー

下部に区切り線とサマリー(状況に応じて色変化)。

### 6.3 ラウンド表示のフォーマットバッジ
各コートヘッダー右側に小型バッジ(`format-tag-small`)を表示し、2v2 / 3v3 を一目で識別できるようにする。

### 6.4 トースト通知
画面下部に短時間(1.6秒)表示する通知 (`#toast`)。リセット系操作で使用。

### 6.5 操作上の確認
プレイヤー削除、リセット、ラウンド取消はすべて `confirm()` ダイアログで再確認。

## 7. セキュリティ / 堅牢性

### 7.1 XSS対策
- ラウンド表示において、プレイヤー名は HTML 文字列に直接埋め込まず、テンプレートで枠を生成後 `textContent` 経由で名前を流し込む二段階レンダリング (`renderRoundDOM`)。
- プレイヤー一覧でも `textContent` を使用。

### 7.2 入力バリデーション
- プレイヤー名: trim 後の空文字を拒否。
- コート数: 0〜20 にクランプ、ボタンの `disabled` でも制限。
- localStorage 読み込み時:
  - `players` が配列であること
  - `courtConfig` が object であれば各値 (count1v1 / count2v2 / count3v3) を `clampCourt` で正規化、合計 0 のときは `count3v3=1` に補正
  - `rounds` の各 `match.format` を `"1v1"` / `"2v2"` 以外はすべて `"3v3"` に正規化

### 7.3 例外処理
- `localStorage` の読み書きは try/catch で保護。

## 9. 既知の仕様 / 制約

- **ラウンド番号は再採番しない**: `undoLastRound` で削除しても以後の番号は連続しない可能性がある。
- **試合数のリセットは履歴も消す**: `gamesPlayed` だけでなく `rounds` も全削除。
- **コート順は固定**: 設定上、1v1 のコートが先(`コート 1..count1v1`)、続いて 2v2(`コート count1v1+1..count1v1+count2v2`)、最後に 3v3(`コート ...+1..`)。
- **履歴のスコア計算はフォーマット非依存**: 1v1 / 2v2 / 3v3 で組んだ経験は同じ重みで全形式の組み立てに影響する。これは「人と人」の相性偏りを抑える狙いだが、形式ごとに区別したい場合は `buildPairHistory` の改修が必要。
- **同時編集**: 複数タブで開いた場合の同期は行われない。
- **エクスポート/共有機能は未実装**。
